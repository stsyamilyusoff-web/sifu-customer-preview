Content + Notifications · five screens, one skin, fixed routing, merged feed
Cluster C of the secondary sweep. The notifications inbox and the four content screens, each before and after, brought onto the parent design language that already lives in the app's code, plus the routing fix and one merge. For Hafiz · 26 June 2026.
The inbox and feeds are still on the old look (full-width blue-wash rows, two near-identical detail screens, bare error text). This brings all five onto one language and fixes two real bugs: the notification that goes nowhere (a tutor-request notification routes to a tab name that does not exist, so the tap silently fails) now navigates, and the two duplicate content surfaces (News and Blogs, whose detail screens differ only by image size and a field name) merge into one "News & Articles" feed sharing one detail screen. Each screen below is its own before/after pair. All zero-backend.
1 · Notifications
The inbox, grouped Recent / Older. Today rows are full-width with a heavy blue wash, and two taps lead nowhere: a tutor-request notification routes to BottomNav > "TutorRequest", a tab that does not exist (the tab is Requests), and any unrecognised sender falls through with no handler. The redesign uses inset rows with a quiet unread dot, fixes the request route, and adds a safe fallback so every tap lands.

Notes for Hafiz
Zero-backend: the whole cluster ships against today's data. The routing fix is a client navigation change (correct tab name plus a fallback branch), the merge is a client consolidation of two screens that already read similar fields, and the re-skin is token-only. No backend dependency.
One thing to confirm: News and Blogs come from two endpoints (NEWS and BLOGS). The merged feed simply calls both and tags each item by source, so no API change is needed. If a single combined endpoint is ever preferred, that is a later optimisation, not a blocker.
